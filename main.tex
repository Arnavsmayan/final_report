{\fontsize{12}{14}\selectfont
Agentic artificial intelligence systems are increasingly being deployed across enterprise workflows for tasks such as document summarization, information extraction, code generation, and decision support. Evaluating whether these systems consistently produce reliable, safe, and goal aligned outputs remains a significant challenge because their behavior is probabilistic, multi step, and highly context dependent.

This project focused on the design and implementation of Agent Eval, a reusable Software Development Kit for evaluating the outputs of agentic workflows using a Large Language Model based judging approach. The SDK provides a single function entry point that accepts the input and output of any agent and returns a structured evaluation result including a numerical score, a pass or fail verdict, governance findings, and detailed reasoning.

Agent Eval is built on a three agent LangGraph state machine. The first agent generates evaluation criteria from the provided input and output, the second agent runs a strict governance check across fifteen evaluators organized into five pillars, and the third agent synthesizes the final score and verdict. The SDK also supports multi model consensus evaluation, where the same workflow is executed independently across multiple judge models and a final synthesis agent produces a consolidated result.

The implemented framework integrates with the internal Eli Lilly LLM Gateway through Microsoft OAuth2 authentication and supports both GPT 5 and Claude Sonnet models. Token usage and cost tracking are built into every evaluation, and a self contained HTML report is automatically generated for each run. The SDK has been validated across a wide variety of evaluation scenarios involving diverse agent inputs and outputs of differing complexity, length, and domain.

The resulting solution provides a vendor independent, extensible, and production ready foundation for governance focused evaluation of agentic AI workflows, enabling teams to gain confidence in agent behavior before production deployment.

\chapter{Introduction}

Agentic artificial intelligence systems have rapidly moved from research prototypes into real production workflows. These systems use large language models to plan, reason, call tools, and produce structured outputs for tasks such as document summarization, information extraction, code generation, and customer support. While the capabilities of these systems continue to grow, the question of how to reliably evaluate their outputs remains largely unsolved. Traditional software testing assumes deterministic behavior, whereas agentic outputs are probabilistic, context dependent, and often free form text.

During my internship at Eli Lilly, I worked on building Agent Eval, a Software Development Kit designed to evaluate the outputs of agentic workflows using a Large Language Model based judging approach. The goal of the project was to provide engineering teams with a simple, consistent, and governance aware way to score agentic outputs before deploying them into production systems.

A key challenge during the project was that existing open source evaluation libraries did not align with the organization's requirements. They lacked governance focused evaluators, were not integrated with the internal LLM Gateway, and did not support multi model consensus evaluation. To address these gaps, a custom SDK was designed and implemented from the ground up with a focus on usability, governance, and operational readiness.

The solution involved designing a three agent evaluation pipeline using LangGraph, defining a fixed catalog of fifteen governance evaluators across five pillars, implementing multi model consensus, integrating with the internal LLM Gateway, and generating self contained HTML reports for every evaluation. Technologies such as Python, LangGraph, LangChain, and the internal Eli Lilly LLM Gateway were leveraged to build a reusable and production ready solution.

The resulting SDK provides a structured way to assess agentic outputs, accelerates evaluation cycles, supports governance discussions, and establishes a strong foundation for further automation of agent quality assurance within the organization. Beyond the technical implementation, the project offered valuable exposure to applied artificial intelligence engineering, system design, and modern software development practices used in enterprise environments.

\chapter{Background}

\section{LLM as Judge in Modern AI Systems}

Modern enterprise applications frequently involve large language models that generate free form, complex, or multi step outputs. Evaluating these outputs is fundamentally different from evaluating traditional software. There is rarely a single correct answer, the output may be valid in multiple forms, and the quality depends on factors such as factual accuracy, reasoning soundness, policy compliance, and contextual relevance.

To address this evaluation gap, the LLM as judge approach has emerged as a widely accepted technique. In this approach, one or more language models are used to assess the quality of outputs produced by another model or agent. This pattern enables scalable, structured, and explainable evaluation that goes beyond keyword matching or static rules.

\section{Existing Evaluation Landscape}

At the start of the project, several open source evaluation libraries were available, including frameworks for retrieval augmented generation evaluation, simple LLM scoring, and general purpose testing utilities. While these tools offered useful starting points, they did not fully meet the requirements of the organization. They lacked governance focused evaluators such as prompt injection detection, data exposure checks, and tool necessity analysis. They also did not integrate with the internal Eli Lilly LLM Gateway or support a structured multi model consensus workflow.

As a result, diagnosing failures in agentic workflows often required ad hoc review by engineers, leading to inconsistent quality assessments, slow feedback cycles, and limited confidence when promoting agentic features to production.

\section{Proof of Concept and Technology Evaluation}

Before committing to a final architecture, a dedicated Proof of Concept (PoC) was developed in an isolated environment. The PoC was used to validate the feasibility of an LLM as judge pipeline, evaluate orchestration frameworks, and explore different evaluator structures before committing to a production design.

The primary objective of the PoC was to evaluate orchestration frameworks for chaining multiple judge agents and to validate the feasibility of generating structured, machine readable scores from natural language judge outputs. During this phase, multiple technologies and architectural patterns were explored, including LangChain, LangGraph, and direct API calls without orchestration libraries.

The evaluation focused on integration requirements, error handling, observability of internal pipeline state, and the ability to extend the design to support multi model consensus. LangGraph was ultimately selected because of its native support for state machine based agent orchestration, predictable execution order, and clean separation between agent responsibilities.

In addition to evaluating orchestration frameworks, commercially available evaluation platforms were also studied to understand their capabilities, integration requirements, and operational implications. While these solutions offered several advanced features, their adoption would have introduced additional licensing, onboarding, and long term operational costs along with vendor lock in concerns.

The evaluation ultimately led to the selection of a custom SDK architecture that maximized reuse of internal organizational capabilities while minimizing additional dependencies and operational costs.

The PoC also helped identify potential implementation challenges and provided valuable insights into how judge outputs could be parsed, validated, and aggregated into a consistent scoring scheme. The findings from these experiments established a clearer understanding of project requirements and enabled informed technology selection for the final solution.

In addition to technical validation, the PoC served as an important communication tool between the engineering team and stakeholders. By demonstrating practical evaluation use cases on real internal data, it enabled stakeholders to visualize the proposed solution, provide early feedback, and align on the SDK's scope and roadmap before final implementation.

\section{Evolution of Project Requirements}

The project initially began with the objective of providing a simple LLM as judge function for scoring agent outputs against a single set of generated criteria. As discussions progressed with engineering teams and stakeholders, the scope expanded significantly.

Additional requirements emerged around governance specific checks, multi model consensus to reduce single model bias, cost tracking, interactive HTML reports, and a clean Python API that could be embedded inside automated workflows. The project gradually evolved from a basic scoring function into a complete evaluation SDK capable of supporting governance, debugging, and analytical use cases.

\section{Motivation for the Project}

The motivation behind this project extended beyond building another evaluation library. It presented an opportunity to work on the practical question of how to safely deploy agentic AI systems within an enterprise environment. Since agentic outputs increasingly drive downstream decisions and business workflows, having a structured and reproducible way to evaluate them is essential.

Unlike traditional feature development, the project required understanding agent behavior, evaluation methodology, prompt engineering for judge agents, and the operational realities of running large language model workloads within a regulated organization. The potential organizational impact of having a unified evaluation framework served as a strong source of motivation. A successful SDK would not only improve confidence in agent outputs but also establish a long term foundation for governance, observability, and quality assurance across applied AI initiatives.

\chapter{Objectives}

The primary objective of this project was to design and implement a reusable Software Development Kit capable of evaluating the outputs of agentic AI workflows using a Large Language Model based judging approach. To achieve this goal, the following objectives were defined:

\begin{enumerate}

\item \textbf{Evaluation of Agentic AI Evaluation Approaches.}
To study and evaluate different agentic evaluation approaches and orchestration frameworks through a dedicated Proof of Concept (PoC), and identify an architecture suitable for the organization's requirements.

\item \textbf{Development of a Reusable Evaluation SDK.}
To design and implement a reusable Python SDK with a single function entry point capable of evaluating the outputs of any agentic workflow, regardless of the underlying agent framework, model, or domain.

\item \textbf{Implementation of a Governance Focused Evaluator Catalog.}
To define and implement a fixed catalog of fifteen governance evaluators organized into five pillars covering goal alignment, compliance, reasoning, tool safety, and robustness.

\item \textbf{Support for Multi Model Consensus Evaluation.}
To enable evaluation using multiple judge models in parallel and synthesize a consensus verdict in order to reduce single model bias and increase confidence in the final score.

\item \textbf{Integration with Internal Gateway and Interactive Reporting.}
To integrate the SDK with the internal Eli Lilly LLM Gateway using Microsoft OAuth2 authentication and to generate a self contained interactive HTML report for each evaluation, including scores, verdicts, evidence, and cost information.

\end{enumerate}

\section{Final Architecture Design}

Based on the insights gained during the Proof of Concept phase, a custom three agent evaluation architecture was designed. The architecture follows a state machine driven approach in which the evaluation input flows through three sequential judge agents, with each agent producing structured output that is consumed by the next stage.

The major components of the architecture and their responsibilities are summarized in Table~\ref{tab:architecture-components}.

\begin{table}[h]
\centering
\caption{Major Components of Agent Eval}
\label{tab:architecture-components}
\begin{tabular}{|p{6cm}|p{10cm}|}
\hline
\textbf{Component} & \textbf{Responsibility} \\
\hline

Child Agent &
Generates evaluation criteria from the provided input and output. Defines what a good output should look like before any scoring takes place.
\\
\hline

Tertiary Agent &
Runs the strict governance check across fifteen evaluators organized into five pillars. Identifies violations, evidence, and risk levels.
\\
\hline

Main Agent &
Synthesizes the final evaluation result by combining the criteria scores and the governance findings into a single numerical score and verdict.
\\
\hline

Final Comparison Agent &
Used only in multi model evaluation. Compares the independent results produced by each model and produces a consensus verdict.
\\
\hline

LillyGatewayLLM Client &
Manages authentication with the internal LLM Gateway using Microsoft OAuth2 and provides a unified interface for invoking GPT 5 and Claude Sonnet.
\\
\hline

Cost Tracker &
Tracks input and output tokens for every step of the pipeline and estimates the corresponding cost based on configurable per model pricing.
\\
\hline

HTML Report Generator &

\section{Architectural Decisions and Customizations}

While several open source evaluation libraries provide basic LLM as judge capabilities, the requirements of this project extended beyond the functionality offered by default implementations. As a result, multiple architectural decisions were introduced to align the SDK with organizational requirements and the internal model infrastructure.

One of the key design decisions involved using LangGraph instead of a plain function call chain. LangGraph enabled predictable execution order, clean state propagation between agents, and a straightforward extension path for adding new agents in the future without rewriting existing code.

Several custom mechanisms were also implemented to improve reliability and developer experience. These included robust JSON parsing for LLM outputs, automatic handling of markdown code fences returned by the model, interactive prompts for missing parameters, defensive validation of inputs, and a fully self contained HTML report that could be opened without any external dependencies.

These customizations enabled the SDK to address organization specific evaluation requirements while maintaining scalability, flexibility, and operational simplicity.

\section{Gateway Integration and Authentication}

All large language model invocations within the SDK are routed through the internal Eli Lilly LLM Gateway. The gateway provides a centrally governed access layer, ensures that only approved models are used, and enforces organizational policies on token usage and access control.

To interact with the gateway, the SDK includes a thin client wrapper that handles token acquisition, request signing, and response normalization. Authentication is performed using the Microsoft OAuth2 client credentials flow, where a short lived access token is obtained from the Microsoft identity platform using the application's client identifier, client secret, and tenant identifier. The token is then attached to every gateway request and refreshed automatically when it expires.

Two model identifiers are supported through the gateway integration:

\begin{itemize}
    \item gpt5, mapped to a GPT 5 model deployment.
    \item claude, mapped to a Claude Sonnet model deployment.
\end{itemize}

This abstraction allows the rest of the SDK to interact with any approved model without having to manage authentication, retry behavior, or vendor specific request formats. It also makes it straightforward to extend the SDK to new models in the future as additional deployments become available through the gateway.

\section{Evaluator Implementation}

The governance evaluator catalog was implemented as a fixed set of fifteen evaluators organized into five pillars. Each evaluator is defined by a name, a short description, a pillar assignment, and a scoring rubric that the judge model follows during evaluation.

The catalog was designed to capture the most important governance dimensions of agentic outputs:

\begin{itemize}
    \item Goal Alignment, covering role adherence, goal drift, and objective completion.
    \item Compliance, covering policy violation and guardrail checks.
    \item Reasoning, covering hallucination, logical consistency, and counterfactual handling.
    \item Tool Safety, covering prompt injection, data exposure, and tool necessity.
    \item Robustness, covering determinism, output consistency, and variance analysis.
\end{itemize}

Special attention was given to ensuring that each evaluator produced a structured response with a score, a passed flag, and reasoning, so that downstream aggregation could be performed in a deterministic way. The evaluator design was kept declarative so that adding or modifying evaluators in the future would require minimal changes to the surrounding pipeline.

\section{Pipeline Execution and Scoring}

Once the SDK receives an input and output, the evaluation request is forwarded through the three agent pipeline. The child agent produces a set of criteria specific to the provided input and output. The tertiary agent runs the governance evaluator catalog and produces per evaluator scores along with violations and evidence. The main agent then synthesizes a final score and verdict based on both the criteria scores and the governance findings.

The execution pipeline ensures that evaluation requests remain structured, reproducible, and easy to inspect. Each step writes its output into a shared state object, which is later used both for the final scoring step and for generating the interactive report.

\section{Multi Model Consensus}

Multi model consensus evaluation was implemented to reduce single model bias and increase confidence in the final verdict. When the SDK is invoked with a list of models, the full three agent pipeline is executed independently for each model.

A separate final comparison agent then compares the per model results and produces a consensus verdict:

\begin{itemize}
    \item Independent execution of the full pipeline for each model.
    \item Comparison of criteria scores and governance findings across models.
    \item Identification of points of agreement and disagreement.
    \item Weighted averaging based on reasoning quality.
    \item Generation of a final consensus verdict and confidence label.
\end{itemize}

The integration of consensus evaluation with the rest of the SDK ensures that users gain access to advanced multi model judging without having to manage orchestration logic themselves.

\section{Cost and Token Tracking}

Token usage and cost tracking are integrated into every evaluation. A dedicated cost tracker is created at the start of each evaluation run and is invoked at each stage of the pipeline. For every step, the tracker estimates the number of input and output tokens consumed and computes the corresponding cost based on per model pricing configured through environment variables.

The cost tracker maintains a per stage breakdown alongside an overall total. At the end of the evaluation, the tracker is serialized into the final result object and surfaced in the generated HTML report. This provides users with full visibility into the operational expense of every evaluation, supports cost analysis when running large evaluation batches, and helps teams reason about whether multi model consensus is justified for a given use case.

By embedding cost tracking directly into the SDK rather than relying on external billing dashboards, engineers receive immediate feedback on token consumption as soon as an evaluation completes. This was particularly valuable during development, where comparing the cost profile of different evaluator configurations informed both prompt design and default pipeline behavior.

\section{Deployment and Validation}

Following the implementation of the SDK, the completed framework was packaged as a Python wheel and integrated with the target evaluation workflows. Comprehensive validation activities were conducted to ensure the reliability, accuracy, and effectiveness of the overall solution.

The validation process focused on verifying the following aspects:

\begin{itemize}
    \item Correctness of criteria generation across diverse inputs.
    \item Reliability of governance evaluator scoring.
    \item Consistency of final score aggregation.
    \item Correctness of multi model consensus behavior.
    \item Stability of cost tracking and report generation.
\end{itemize}

Testing was performed across multiple scenarios involving diverse agent inputs and outputs of varying complexity, length, and domain. The validation set was designed to cover both well behaved outputs and intentionally degraded outputs in order to confirm that the scoring pipeline reacted appropriately and that violations were surfaced with accurate evidence. Additional validation ensured that the generated HTML report rendered correctly across different browsers and operating systems.

Successful validation confirmed that the SDK was capable of providing reliable, structured, and reproducible evaluations of agentic outputs while maintaining a clean developer experience.

\chapter{Limitations}

While the implemented Agent Eval SDK provides comprehensive evaluation of agentic outputs, certain limitations remain within the scope of the current implementation.

The solution has been designed and validated primarily for text based agent outputs. Extending the same evaluation framework to multi modal outputs such as images, audio, or video would require additional research, format specific evaluators, and integration with model specific judging capabilities.

Although the SDK captures rich governance findings, score breakdowns, and cost information, it does not currently provide longitudinal tracking of evaluations across runs or detailed per agent regression analysis. Advanced capabilities such as persistent evaluation history, cross run comparison, and trend dashboards were intentionally kept outside the scope of this project.

The evaluation quality available through the SDK is bounded by the capabilities and biases of the underlying judge models. Any limitations or blind spots present in the judge models may carry over into the evaluation results, particularly in highly specialized domains.

Furthermore, the effectiveness of governance evaluators depends on the quality of the prompts and rubrics used to drive them. Any uncovered scenarios or under specified rubrics may reduce the depth of analysis for certain types of outputs.

Despite these limitations, the implemented solution successfully achieves the primary objective of providing structured, governance aware, and reproducible evaluation of agentic outputs. The SDK establishes a strong foundation for future enhancements and expansion of evaluation capabilities.

\chapter{Conclusion and Future Work}

\section{Conclusion}

The project successfully achieved its objective of designing and implementing a reusable SDK for evaluating agentic AI outputs at Eli Lilly. Through a structured approach involving requirement analysis, proof of concept development, architecture design, and production implementation, a scalable and vendor independent evaluation solution was established.

The implemented SDK provides structured visibility into criteria scores, governance findings, violations, evidence, and cost information. By integrating criteria generation, governance evaluation, final synthesis, and reporting into a unified framework, the solution enables faster review, more consistent quality assessment, and stronger confidence before deploying agentic features to production.

Beyond the technical implementation, the project provided valuable experience in applied AI engineering, system design, prompt engineering for judge agents, and cross functional collaboration. The successful packaging of the SDK as a reusable Python wheel establishes a strong foundation for future evaluation initiatives and contributes to improving the reliability, governance, and maintainability of agentic AI workflows within the organization. The project also demonstrated that enterprise grade evaluation can be achieved through a carefully designed, vendor independent architecture that leverages existing organizational infrastructure, thereby reducing both operational complexity and long term cost.

\section{Future Work}

Although the current implementation fulfills the primary project objectives, several enhancements can be explored in the future:

\begin{itemize}
\item Extension of the evaluation framework to multi modal agent outputs through format specific evaluators and judging strategies.

\item Implementation of persistent evaluation history, cross run comparison, and regression analysis to support longitudinal quality tracking.

\item Integration with continuous integration pipelines to enable automatic evaluation gates for agentic features.

\item Development of adaptive evaluator selection based on the structure and content of the agent output.

\item Expansion of domain specific evaluators to support specialized use cases such as clinical, regulatory, and scientific workflows.

\end{itemize}

These enhancements can further strengthen the evaluation ecosystem and provide deeper insights into agent performance, reliability, and safety. The current implementation serves as a scalable foundation upon which future evaluation and governance capabilities can be continuously expanded and refined.
