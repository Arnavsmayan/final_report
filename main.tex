\vspace{0.05in}

{\fontsize{12}{14}\selectfont
Agentic artificial intelligence systems are increasingly deployed across enterprise workflows for tasks such as document summarization, information extraction, code generation, and decision support. Evaluating whether these systems consistently produce reliable, safe, and goal aligned outputs remains challenging because their behavior is probabilistic, multi step, and highly context dependent.

This project focused on the design and implementation of Agent Eval, a reusable Software Development Kit for evaluating the outputs of agentic workflows using a Large Language Model based judging approach. The SDK provides a single function entry point that accepts the input and output of any agent and returns a structured evaluation result including a numerical score, a pass or fail verdict, governance findings, and detailed reasoning.

Agent Eval is built on a three agent LangGraph state machine. A child agent generates evaluation criteria, a tertiary agent runs a strict governance check across fifteen evaluators organized into five pillars, and a main agent synthesizes the final score and verdict. The SDK also supports multi model consensus, where the pipeline is executed independently across multiple judge models and a final synthesis agent produces a consolidated result.

The framework integrates with the internal Eli Lilly LLM Gateway and supports both GPT 5 and Claude Sonnet. Token usage and cost tracking are built in, and a self contained HTML report is generated for every run. The resulting solution provides a vendor independent, extensible, and production ready foundation for governance focused evaluation of agentic AI workflows.

\chapter{Introduction}

Agentic AI systems have moved rapidly from research prototypes into production workflows for tasks such as summarization, extraction, code generation, and support. Their outputs, however, are probabilistic and free form, so traditional deterministic testing cannot reliably assess them. Available open source evaluation libraries also lacked governance focused checks, LLM Gateway integration, and multi model consensus support required inside the organization.

To close these gaps, this internship delivered \textbf{Agent Eval}: a Python SDK that runs a three agent LangGraph pipeline, applies fifteen governance evaluators across five pillars, supports multi model consensus, integrates with the internal Eli Lilly LLM Gateway, and produces a self contained interactive HTML report for every evaluation. The result is a single reusable entry point that engineering teams can drop into any agentic workflow to obtain a structured, reproducible, and governance aware quality assessment.

\chapter{Background}

\section{LLM as Judge}

Enterprise applications increasingly rely on LLMs that produce free form or multi step outputs. These outputs rarely have a single correct answer and depend on factors such as factual accuracy, reasoning soundness, and policy compliance. The LLM as judge pattern~\cite{zheng2023judging, liu2023geval}, in which one model assesses the quality of another model's output, has emerged as a scalable and explainable alternative to keyword matching or static rules.

\section{Existing Evaluation Landscape}

Available open source evaluation libraries offered useful primitives but did not meet the organization's needs: they lacked governance focused evaluators such as prompt injection detection, data exposure checks, and tool necessity analysis, they did not integrate with the internal LLM Gateway, and they did not support structured multi model consensus. In the absence of a suitable tool, failures in agentic workflows were diagnosed through ad hoc engineer review, producing inconsistent assessments and slow feedback cycles.

\section{Proof of Concept}

A dedicated PoC was built in an isolated environment to validate the feasibility of an LLM as judge pipeline before committing to a production design. The PoC focused on:

\begin{itemize}
    \item Evaluating orchestration frameworks for chaining multiple judge agents.
    \item Validating parsing of structured JSON responses from LLM outputs.
    \item Studying state propagation between agents across a multi step pipeline.
    \item Exploring evaluator structures suitable for governance focused checks.
    \item Comparing single model and multi model evaluation behavior.
\end{itemize}

The frameworks compared were LangChain~\cite{langchain}, LangGraph~\cite{langgraph}, and direct API calls without orchestration. LangGraph was selected for its native state machine orchestration, predictable execution order, and clean separation between agent responsibilities. Commercial evaluation platforms were also reviewed but ruled out due to licensing cost, onboarding overhead, and vendor lock in, leaving a custom SDK as the preferred direction.

\chapter{Objectives}

The following objectives guided the design and implementation of the SDK:

\begin{enumerate}

\item \textbf{Evaluation of Agentic AI Evaluation Approaches.}
To study and evaluate different agentic evaluation approaches and orchestration frameworks through a dedicated Proof of Concept (PoC), and identify an architecture suitable for the organization's requirements.

\item \textbf{Development of a Reusable Evaluation SDK.}
To design and implement a reusable Python SDK with a single function entry point capable of evaluating the outputs of any agentic workflow, regardless of the underlying agent framework, model, or domain.

\item \textbf{Implementation of a Governance Focused Evaluator Catalog.}
To define and implement a fixed catalog of fifteen governance evaluators organized into five pillars covering goal alignment, compliance, reasoning, tool safety, and robustness.

\item \textbf{Support for Multi Model Consensus Evaluation.}
To enable evaluation using multiple judge models in parallel and synthesize a consensus verdict in order to reduce single model bias and increase confidence in the final score.

\item \textbf{Integration with Internal Gateway and Interactive Reporting.}
To integrate the SDK with the internal Eli Lilly LLM Gateway using Microsoft OAuth2 authentication~\cite{msoauth2} and to generate a self contained interactive HTML report for each evaluation, including scores, verdicts, evidence, and cost information.

\end{enumerate}

\section{Requirement Analysis}

The project began with discussions involving applied AI engineers, platform teams, and stakeholders to identify gaps in existing evaluation tooling. The categories of agentic workflows considered included:

\begin{itemize}
    \item Document summarization pipelines.
    \item Structured information extraction agents.
    \item Code generation and refactoring assistants.
    \item Chatbot and question answering systems.
\end{itemize}

The planned deliverable was a distributable Python package~\cite{pypa} with strict Pydantic~\cite{pydantic} runtime validation of both configuration inputs and LLM outputs, engineered to integrate cleanly with the internal LLM Gateway.

\section{Proof of Concept Development}

A separate PoC environment was used before any production commitment. It focused on:

\begin{itemize}
    \item Evaluating orchestration frameworks for chaining judge agents.
    \item Validating structured JSON parsing from LLM outputs.
    \item Studying state propagation between agents across pipeline stages.
    \item Comparing single model and multi model evaluation behavior.
    \item Assessing extensibility for future evaluators.
\end{itemize}

These experiments established the technical foundation and confidence needed before moving toward a production ready implementation.

\section{Final Architecture}

Building on the PoC, a three agent, state machine driven architecture was adopted. The evaluation input flows sequentially through three judge agents, each producing structured output consumed by the next. \subsection{Component Architecture}

Figure~\ref{fig:component_arch} shows the module level view: the public entry point, parameter validation layer, gateway client, three agents that form the single model pipeline, and the final comparison agent used only in multi model runs.

\begin{figure}[h]
    \centering
    \includegraphics[width=\textwidth]{component_architecture.png}
    \caption{Component architecture of the Agent Eval SDK.}
    \label{fig:component_arch}
\end{figure}

The internal responsibilities of each component are:

\begin{itemize}
    \item \textbf{\texttt{evaluate()} entry point.} Validates parameters, routes between single and multi model paths, and triggers dashboard generation.
    \item \textbf{Validation layer.} Checks credentials, prompts for missing arguments, verifies the pass threshold, and confirms that requested models and evaluators exist.
    \item \textbf{\texttt{LillyGatewayLLM} client.} Handles OAuth2 authentication, resolves model identifiers to gateway deployments, exposes raw and JSON aware invocation, and refreshes tokens transparently.
    \item \textbf{Child agent.} Generates evaluation criteria from the input and output before any scoring takes place.
    \item \textbf{Tertiary agent.} Runs the full governance evaluator catalog in strict mode and returns per evaluator scores, violations, and evidence.
    \item \textbf{Main agent.} Computes per criterion scores and synthesizes the final score and verdict.
    \item \textbf{Final comparison agent.} Invoked only when \texttt{model} is a list: weights each participating model by reasoning quality and computes a consensus score that is guaranteed to use weights summing to one hundred.
\end{itemize}

\section{Architectural Decisions}

Because default open source implementations did not cover the required scope, several project specific decisions shaped the design:

\begin{itemize}
    \item \textbf{LangGraph over a plain call chain.} Chosen for predictable execution order, clean state propagation, and a straightforward extension path for new agents.
    \item \textbf{Robust JSON parsing.} Tolerates markdown code fences and malformed structures returned by the model.
    \item \textbf{Interactive parameter prompts.} Missing arguments are collected from the console when \texttt{interactive} is enabled.
    \item \textbf{Defensive input validation.} Every argument is checked before any model call is made.
    \item \textbf{Self contained HTML report.} Renders offline with no external CSS, JS, or asset dependencies.
\end{itemize}

\section{Gateway Integration and Authentication}

All LLM invocations are routed through the internal Eli Lilly LLM Gateway, which provides a governed access layer and enforces organizational policies on approved models, token usage, and access control. A thin client wrapper inside the SDK handles:

\begin{itemize}
    \item Token acquisition via the Microsoft OAuth2 client credentials flow~\cite{msoauth2}.
    \item Automatic refresh of short lived access tokens on expiry.
    \item Attachment of the access token to every gateway request.
    \item Model identifier resolution to the correct gateway deployment.
\end{itemize}

Two gateway model identifiers are supported today: \texttt{gpt5} (a GPT 5 deployment) and \texttt{claude} (a Claude Sonnet deployment). Adding further approved deployments requires only registering the identifier and does not affect the rest of the SDK.

\texttt{evaluators} &
The literal string all, the value none, or a list of evaluator or pillar names. &
Selects which governance evaluators are executed. The default value all runs the full catalog of fifteen evaluators across the five pillars.
\\
\hline

\texttt{success\_criteria} &
Optional list of strings. &
Allows the caller to override the criteria generated by the child agent. When omitted, the child agent produces criteria automatically from the input and output.
\\
\hline

\texttt{interactive} &
Boolean. Default is true. &
Controls whether the SDK prompts on the console for any missing required parameter. When false, missing values raise an error instead.
\\
\hline

\end{tabular}
\end{table}

This single function design keeps the SDK surface area small while still supporting workflows that range from quick ad hoc checks to fully governed multi model consensus runs.

\section{Evaluator Implementation}

The governance evaluator catalog is a fixed set of fifteen evaluators organized into five pillars: Goal Alignment (role adherence, goal drift, objective completion), Compliance (policy violation, guardrail check), Reasoning (hallucination, logical consistency, counterfactual), Tool Safety (prompt injection, data exposure, tool necessity), and Robustness (determinism, output consistency, variance analysis). Each evaluator carries a name, description, pillar assignment, and scoring rubric that the judge model follows during evaluation. The design is declarative so future evaluators can be added with minimal changes to the surrounding pipeline.

Every evaluator returns the same five field response structure (Table~\ref{tab:evaluator-output}). This uniform contract lets the pipeline aggregate results, apply thresholds, and render the report without knowing anything evaluator specific.

\section{Multi Model Consensus}

Multi model consensus reduces single model bias and increases confidence in the final verdict. When \texttt{evaluate()} is called with a list of models, the SDK:

\begin{itemize}
    \item Runs the full three agent pipeline independently for each configured model.
    \item Compares criteria scores and governance findings across models.
    \item Identifies points of agreement and disagreement between models.
    \item Assigns percentage weights to each model based on reasoning quality.
    \item Produces the final consensus verdict alongside a confidence label.
\end{itemize}

The multi model nature is reflected throughout the report: the cost panel in Figure~\ref{fig:html_report_scoring} breaks down tokens and cost by GPT-5, Claude, and the consensus agent, and the panels in Figure~\ref{fig:violations_evidence} attribute each finding to its originating judge, so any disagreement between GPT-5 and Claude is preserved rather than hidden.

\section{Cost and Token Tracking}

A cost tracker is instantiated at the start of each run and invoked at every stage of the pipeline. For each stage, the tracker records:

\begin{itemize}
    \item Input tokens consumed by the model call.
    \item Output tokens produced by the model.
    \item Cost derived from per model pricing configured via environment variables.
    \item A per stage breakdown alongside an overall run total.
\end{itemize}

The tracker is serialized into the final result object and surfaced in the HTML report, so engineers receive immediate feedback on token consumption as soon as an evaluation completes. This proved particularly useful when comparing the cost profile of different evaluator configurations during development.

\subsection{Performance Characteristics}

The runtime footprint was profiled during validation to give teams a realistic sense of the cost of adopting the SDK. A single model evaluation issues three sequential model calls (child, tertiary, main) and completes in roughly thirty to sixty seconds, consuming approximately sixty to seventy thousand tokens; the tertiary agent dominates because it runs the full evaluator catalog. A two model configuration issues seven total calls (three per model plus the final comparison agent) and completes in roughly sixty to one hundred twenty seconds, with tokens scaling proportionally. \section{Deployment and Validation}

The completed SDK was packaged as a Python wheel and integrated with the target evaluation workflows. Validation focused on verifying:

\begin{itemize}
    \item Correctness of criteria generation across diverse inputs.
    \item Reliability of governance evaluator scoring.
    \item Consistency of final score aggregation.
    \item Correctness of multi model consensus behavior.
    \item Stability of cost tracking and report generation.
\end{itemize}

Test inputs ranged across domains and complexities and deliberately included degraded outputs to confirm that the pipeline surfaced violations with accurate evidence. Report rendering was verified across major browsers and operating systems. The results confirmed that the SDK delivered reliable, structured, and reproducible evaluations while retaining a clean developer experience.

\chapter{Limitations}

While the implemented SDK provides comprehensive evaluation of agentic outputs, certain limitations remain within the scope of the current implementation:

\begin{itemize}
    \item The SDK covers text based agent outputs; multi modal outputs (images, audio, video) would require format specific evaluators and integration with model specific judging.
    \item Longitudinal capabilities such as persistent evaluation history, cross run comparison, and trend dashboards were intentionally kept out of scope.
    \item Evaluation quality is bounded by the capabilities and biases of the underlying judge models, particularly in highly specialized domains.
    \item The depth of governance analysis depends on the quality of evaluator prompts and rubrics; uncovered scenarios or under specified rubrics may reduce coverage.
\end{itemize}

Despite these limitations, the SDK achieves its primary objective of providing structured, governance aware, and reproducible evaluation, and establishes a solid foundation for future enhancements.

\chapter{Conclusion and Future Work}

\section{Conclusion}

The project delivered a reusable SDK for evaluating agentic AI outputs at Eli Lilly. Structured requirement analysis, a targeted PoC, and a clean three agent architecture yielded a scalable, vendor independent solution that provides structured visibility into criteria scores, governance findings, violations, evidence, and cost, and that packages cleanly as a Python wheel for other engineering teams. Beyond the technical outcome, the internship provided direct experience in applied AI engineering, system design, prompt engineering for judge agents, and cross functional collaboration, and demonstrated that enterprise grade evaluation can be achieved by leveraging existing organizational infrastructure rather than onboarding external platforms.

\section{Future Work}

\begin{itemize}
\item Extension to multi modal agent outputs through format specific evaluators.
\item Persistent evaluation history, cross run comparison, and regression analysis for longitudinal tracking.
\item Integration with CI pipelines to enable automatic evaluation gates for agentic features.
\item Adaptive evaluator selection driven by output structure and content.
\item Expansion of domain specific evaluators for clinical, regulatory, and scientific workflows.
\end{itemize}
